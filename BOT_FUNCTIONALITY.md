# Функционал и логика работы бота ДМК

Структурированное описание по модулям: сценарии, отображаемые данные в уведомлениях, поиске и выводе информации.

---

## 1. Роли и доступ

| Роль            | Доступ |
|-----------------|--------|
| **Администратор** | Пользователи (все операции), заявки на ремонт, рейсы, зарплата |
| **Логист**      | Рейсы (создание, просмотр, поиск по водителю/ТС, удаление, фильтры по статусам) |
| **Бухгалтер**   | Зарплата (загрузка, поиск, не подтверждённые, выгрузка за период, удаление) |
| **Механик**     | Заявки на ремонт (обработка, поиск по ТС, фильтры по статусам) |
| **Водитель**    | Рейсы (принятие, точки, завершение), зарплата (просмотр, подтверждение/комментарий), ремонт (заявка, мои ремонты) |

Подключение роутеров: `handlers/dispatcher.py` — отдельно водительский роутер, общий для не-водителей, роутер только для админа (пользователи).

## 2. Модуль «Пользователи» (только администратор)

**Файлы:** `handlers/admin/users.py`, `keyboards/admin_kb.py` (get_users_keyboard, get_edit_user_*), `states/user_states.py` (RegisterState, BlockedState, EditUserState), `database/repositories/user_repository.py`.

### 2.1 Все пользователи

- **Кнопка:** «👩‍👩‍👦‍👦 Все пользователи».
- **Логика:** выборка по ролям и статусу (активные по роли + отдельно «Не подтвержденные водители»).
- **Формат вывода:** по каждой роли заголовок (Администраторы, Водители, Логисты, Бухгалтеры, Механики, Не подтвержденные водители), затем строки вида:
  - `ID: <tg_id или -> | ФИО: <name> | 📞 <phone или ->`
  - при отсутствии пользователей по роли — строка «  — нет».

### 2.2 Добавить пользователя

- **Кнопка:** «➕ Добавить пользователя».
- **Шаги:** ФИО → телефон → выбор роли (инлайн: Водитель, Администратор, Логист, Бухгалтер, Механик).
- **Результат:** создаётся запись в БД со статусом «Не подтверждён» (tg_id=0). При первом входе в бота по совпадению ФИО пользователь активируется (tg_id и статус обновляются).
- **Ответ после создания:** «Пользователь создан» + ФИО, телефон, должность.

### 2.3 Изменить пользователя

- **Кнопка:** «✏️ Изменить пользователя».
- **Шаги:**
  1. Ввод **ID пользователя (Telegram ID)**. ID берётся из раздела «Все пользователи».
  2. Если пользователь найден — вывод карточки (ФИО, ID, телефон, статус, роль) и **инлайн-меню** выбора параметра:
     - **👤 ФИО**
     - **📞 Телефон**
     - **📊 Статус**
     - **🎯 Роль**
  3. В зависимости от выбора:
     - **ФИО / Телефон:** запрос нового значения текстом → обновление в БД → сообщение «✅ … изменено».
     - **Статус:** инлайн-меню (✅ Активен, ⏳ Не подтверждён, ❌ Заблокирован) → обновление статуса → подтверждение.
     - **Роль:** инлайн-меню (Водитель, Администратор, Логист, Бухгалтер, Механик) → обновление роли → подтверждение.
- **Отображаемые данные при выборе пользователя:** ФИО, ID (tg_id), телефон, статус (Активен / Не подтверждён / Заблокирован), роль (человекочитаемое название).

### 2.4 Заблокировать пользователя

- **Кнопка:** «❌ Заблокировать пользователя».
- **Шаги:** ввод ID (tg_id) → вызов `user_repository.block_user(tg_id)` (установка статуса BLOCKED).
- **Ответ:** «Пользователь с ID … заблокирован» или «не найден».

## 3. Модуль «Рейсы»

**Файлы:** `handlers/admin/routes.py`, `handlers/logistic/routes.py`, `handlers/driver/routes.py`, `database/repositories/route_repository.py`, `database/models/route.py`, `services/notification_service.py` (notify_about_status_point).

### 3.1 Создание рейса (админ / логист)

- **Кнопка:** «➕ Создать рейс» → инлайн: **✏️ Вручную** / **📄 Формат 1С**.

**Ручной формат:** ФИО водителя → номер ТС (с валидацией) → номер рейса → температура (или «-») → контакты диспетчера (или «-») → номер для регистрации (или «-») → номер прицепа (или «-») → создание рейса → добавление точек (тип, дата, место) → «Сохранить рейс».

**Формат 1С:** сразу запрос сообщения в формате 1С (без предварительного ввода ФИО/ТС). Парсинг: номер рейса, номер для регистрации, ФИО водителя, номер ТС, номер прицепа, температура, контакты; блоки Загр/Выгр и Организация (в т.ч. на одной строке). При отсутствии в тексте: **до** создания рейса запрашиваются ФИО водителя (обязательно) и номер ТС (можно пропустить); **после** создания — номер для регистрации и номер прицепа (можно пропустить).

**Данные в уведомлении водителю о новом рейсе** (и при «Сохранить рейс»): номер рейса, ТС, прицеп, температура, контакты, номер для регистрации (если заданы), список точек (тип, место, дата). Если у водителя уже есть активный рейс — текст об этом; иначе — кнопка «✅ Принять рейс».

### 3.2 Удаление рейса

- Ввод номера рейса → показ данных рейса в формате списка (как ниже) → инлайн «Подтвердить удаление» / «Отменить».

### 3.3 Поиск и отображение рейсов

- **Рейсы по водителю:** ввод ФИО → инлайн: Все / Не приняты / В процессе / Завершены → вывод списка.
- **Рейсы по ТС:** ввод номера ТС → то же инлайн по статусам → вывод списка.
- **Кнопки «Не принятые» / «В процессе» / «Завершенные»:** вывод всех рейсов с выбранным статусом.

**Формат вывода списка рейсов (админ/логист):** для каждого рейса — номер рейса, водитель (ФИО), статус (Не принят / В процессе / Завершен), ТС, прицеп, температура, контакты, номер для регистрации (при наличии), перечень точек (тип, место, дата).

### 3.4 Водитель: рейсы

- **Активный рейс:** если есть рейс в статусе «В процессе» — показ текущей точки (тип, дата, место) и кнопок по шагам; в шапке — номер рейса, ТС, прицеп, температура, контакты, номер для регистрации. Если все точки выполнены — список точек и кнопка «Завершить рейс».
- **Список новых рейсов:** номер рейса, ТС, прицеп, температура, контакты, номер для регистрации, количество точек; инлайн выбора рейса для принятия.

### 3.5 Уведомления по рейсам

- **Админам и логистам** (смена статуса точки/рейса водителем): рейс (id), водитель (ФИО), ТС, прицеп, температура, контакты, номер для регистрации, текст действия (выехал на точку, зарегистрировался и т.д.). Функция: `notification_service.notify_about_status_point`.

## 4. Модуль «Зарплата»

**Файлы:** `handlers/admin/salary.py`, `handlers/accountant/salary.py`, `handlers/driver/salary.py`, `services/notification_service.py` (MessageSalary).

### 4.1 Администратор / бухгалтер

- **Загрузить расчет за день:** водитель (ФИО) → одна строка с 37 параметрами (пробел/запятая; десятичные — точка или запятая) → сохранение → уведомление водителю.
- **Найти расчет за день:** водитель → дата → вывод расчёта (все ненулевые поля в едином формате).
- **Не подтвержденные расчеты:** список расчётов без подтверждения водителем.
- **Удалить расчет:** водитель → дата → выбор расчёта по ID → подтверждение.
- **Выгрузить расчет за период:** водитель → тип периода (с начала месяца / месяц / произвольный) → при необходимости границы периода → выдача CSV.

**Формат отображения расчёта:** ID, дата, рейс (если есть); все ненулевые числовые и текстовые поля (в т.ч. паллеты, тарифы, адреса, транспорт, прицеп); в конце — статус (ожидает подтверждения / подтверждено / с комментарием и текст комментария).

### 4.2 Водитель

- **Показать расчет за день:** дата → вывод расчёта в том же формате; при необходимости — инлайн «Подтвердить» / «Комментарий».
- **Не подтвержденные расчеты:** список своих расчётов с кнопками подтверждения/комментария.
- **Выгрузить расчет за период:** выбор периода → CSV.

### 4.3 Уведомления по зарплате

- **Водителю о новом расчёте:** полный формат расчёта (все ненулевые поля) + кнопки подтверждения/комментария. Метод: `MessageSalary.notify_driver_about_salary`.
- **Бухгалтерам и администраторам** при подтверждении водителем: текст «Водитель … подтвердил расчет» + ID, дата, ЗП + полный разбор расчёта. Метод: `notify_accountants_about_salary_confirmation`.
- **Бухгалтерам и администраторам** при комментарии водителя: «Водитель … оставил комментарий» + ID, дата, комментарий + полный разбор расчёта. Метод: `notify_accountants_about_salary_comment`.

## 5. Модуль «Заявки на ремонт»

**Файлы:** `handlers/driver/repairs.py`, `handlers/admin/repairs.py`, `handlers/mechanic/repairs.py`, `services/notification_service.py`.

### 5.1 Водитель

- **Заявка на ремонт:** номер ТС (форматы как в рейсах) → описание неисправности → подтверждение → создание заявки (ID = MAX(id_ticket)+1), уведомление админам и механикам.
- **Мои ремонты:** список своих заявок.

**Уведомление админам/механикам о новой заявке:** номер заявки, номер ТС, описание неисправности, водитель (ФИО), телефон; кнопка «Обработать». Метод: `notify_about_new_repair`.

### 5.2 Администратор / механик

- **Обработать заявку:** выбор заявки (инлайн) → ввод даты, места, комментария → смена статуса, уведомление водителю.
- **Найти заявки ТС:** ввод номера ТС → список заявок по этому ТС.
- Кнопки по статусам: Новые, Ожидают подтверждения, Ожидают ремонта, Выехали на ремонт, Отремонтированы, Завершены — вывод списка заявок в выбранном статусе.

**Уведомления:**
- Водителю при обработке заявки админом/механиком — по сценарию обработки (дата, место, комментарий и т.д.).
- Админам и механикам при подтверждении заявки водителем: номер заявки, водитель, телефон, ТС, неисправность, дата/место/комментарий ремонта, статус. Метод: `notify_about_confirmed_repair`.
- При выезде на ремонт: те же поля + статус «Выехал на ремонт». Метод: `notify_about_departure_for_repair`.
- При завершении ремонта водителем: те же поля + «Подтвердите окончание ремонта» и кнопка. Метод: `notify_about_completed_repair`.

## 6. Общие элементы

### 6.1 Старт и авторизация

- **Файл:** `handlers/common/start_handlers.py`. По `/start` или «Начать» — проверка роли; для неприглашённых запрос ФИО и сопоставление с БД (активация при совпадении).
- **Назад:** `handlers/common/back_handler.py` — кнопки «↩️ Назад» / «🔙 Назад» возвращают в главное меню или в меню раздела.

### 6.2 База данных (основное)

- **Users:** tg_id, name, phone, role, status (active / invite / blocked).
- **Route:** id, tg_id, points, status (new / process / success), number_auto, temperature, dispatcher_contacts, registration_number, trailer_number.
- **Point:** id, id_route, type_point, place_point, date_point, статусы времени и фото, status.
- **Salary:** поля расчёта ЗП (модель и репозиторий с полным набором полей).
- **Repair:** id_ticket, tg_id, number_auto, malfunction, дата/место/комментарий ремонта, статусы.

### 6.3 Сводка: где что выводится

| Модуль      | Контекст                    | Отображаемые данные |
|-------------|-----------------------------|----------------------|
| Пользователи | Все пользователи            | По ролям: ID, ФИО, телефон |
| Пользователи | Изменить пользователя       | Карточка: ФИО, ID, телефон, статус, роль; после изменения — подтверждение |
| Рейсы       | Списки (админ/логист)       | Рейс, водитель, статус, ТС, прицеп, температура, контакты, номер для регистрации, точки |
| Рейсы       | Уведомление водителю        | Номер рейса, ТС, прицеп, температура, контакты, номер для регистрации, точки, кнопка «Принять рейс» |
| Рейсы       | Уведомления админ/логист    | Рейс, водитель, ТС, прицеп, температура, контакты, номер для регистрации, статус точки |
| Рейсы       | Водитель (активный/новые)   | Те же поля рейса + точки/статусы точек |
| Зарплата    | Поиск / показ расчёта       | Все ненулевые поля расчёта, статус, комментарий |
| Зарплата    | Уведомления (новый расчёт, подтверждение, комментарий) | Полный разбор расчёта + контекст сообщения |
| Ремонт      | Уведомления по заявкам      | Номер заявки, водитель, телефон, ТС, неисправность, дата/место/комментарий, статус |
