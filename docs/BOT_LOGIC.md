# Логика работы телеграм-бота ДМК

Документ описывает модули бота, сценарии и формат выводимой информации.

---

## 1. Роли и доступ

- **Администратор** — полный доступ: пользователи, ремонты, рейсы, зарплата.
- **Логист** — рейсы (создание, просмотр, поиск), без раздела пользователей.
- **Бухгалтер** — зарплата (загрузка, поиск, не подтверждённые, выгрузка за период, удаление).
- **Механик** — заявки на ремонт (обработка, поиск по ТС, фильтры по статусам).
- **Водитель** — рейсы (принятие, выполнение точек, завершение), зарплата (просмотр, подтверждение/комментарий), заявки на ремонт (создание, мои ремонты).

Роутеры подключаются через `handlers/dispatcher.py`: отдельно водительский роутер и общий для не-водителей (админ, логист, бухгалтер, механик), плюс роутер только для админа (пользователи).

---

## 2. Пользователи (только администратор)

- **Все пользователи** — список по ролям: Администраторы, Логисты, Бухгалтеры, Механики, Водители. Для каждого: `ID: <tg_id>`, ФИО, телефон. Если по роли никого нет — «— нет».
- **Добавить пользователя** — запрос ФИО → телефон → выбор роли (инлайн). Создаётся пользователь со статусом «Не подтверждён». После первого входа в бота по совпадению ФИО пользователь переводится в активные.
- **Заблокировать** — запрос tg_id пользователя для блокировки.

**Формат вывода «Все пользователи»:** заголовок по роли, затем строки вида `ID: 123456789  ФИО  Телефон`.

## 3. Рейсы

### 3.1 Создание рейса (администратор / логист)

**Кнопка «➕ Создать рейс»** — сначала выбор формата (инлайн): **Вручную** или **Формат 1С**.

#### Ручной формат

1. ФИО водителя (поиск пользователя по имени).
2. Номер ТС (валидация: А111АА222, А111АА22, ЕН068477 и аналоги).
3. Номер рейса (уникальный).
4. Температурный режим (или «-» для пропуска).
5. Контакты диспетчера (или «-»).
6. Номер для регистрации (или «-»).
7. Номер прицепа (или «-»).
8. Создание рейса → переход к добавлению точек (инлайн: «Добавить точку» / «Сохранить рейс»).

Добавление точки: тип (Загрузка/Разгрузка) → дата → место. После «Сохранить рейс» водителю отправляется уведомление о новом рейсе.

#### Формат 1С

1. ФИО водителя (если не будет в тексте 1С).
2. Номер ТС (если не будет в тексте 1С).
3. Сообщение в формате 1С (одним блоком).

**Ожидаемый формат сообщения 1С (порядок и наличие полей могут меняться):**

- Первая строка без двоеточия или строка с номером — **номер рейса**.
- Строки вида «Ключ: значение»:
  - **Номер для регистрации:** число/строка.
  - **ФИО водителя:** текст (используется для поиска водителя, если не был введён ранее).
  - **Номер ТС:** текст.
  - **Номер прицепа:** текст.
  - **Температура:** текст (например «нет», «+3 С»).
  - **Контакты ООО "ДМК":** или **Контакты:** — контакты логиста/диспетчера.
- Далее блоки точек:
  - **Загр:** дата и время (всё после двоеточия).
  - Следующая строка **Организация:** или **Организация** без двоеточия — адрес точки.
  - **Выгр:** дата и время.
  - **Организация** / **Организация:** — адрес.

Если в разборе не найдены **номер для регистрации** или **номер прицепа**, после создания рейса и точек бот по очереди запрашивает недостающие значения; можно отправить «-» и пропустить. После последнего недостающего параметра рейс сохраняется и водителю отправляется уведомление.

**Вывод после создания рейса из 1С:** сообщение «Рейс создан по формату 1С» + номер рейса, водитель, ТС, прицеп (если есть), температура, контакты, номер для регистрации (если есть), количество созданных точек.

### 3.2 Удаление рейса

- **Удалить рейс** → ввод номера рейса → показ данных рейса и инлайн «Подтвердить удаление» / «Отменить». При подтверждении удаляются рейс и все его точки.

**Формат показа рейса перед удалением:** как в общем списке рейсов (см. ниже), плюс вопрос о подтверждении.

### 3.3 Поиск рейсов

- **Рейсы по водителю** — ввод ФИО водителя → инлайн-меню: **Все**, **Не приняты**, **В процессе**, **Завершены**. По выбору выводится список рейсов с выбранным статусом (или все рейсы водителя).
- **Рейсы по ТС** — ввод номера ТС → такое же инлайн-меню по статусам → вывод списка рейсов по выбранному критерию.

**Формат списка рейсов (админ/логист):** заголовок (например «Все рейсы» или «Рейсы водителя X (Не принятые)»). Для каждого рейса:
- Рейс: \<id\>
- Водитель: \<ФИО\>
- Статус: Не принят / В процессе / Завершен
- ТС: \<номер\> (если есть)
- Прицеп: \<номер\> (если есть)
- Температура: \<значение\> (если есть)
- Контакты: \<значение\> (если есть)
- Номер для регистрации: \<значение\> (если есть)
- Список точек: тип (Загрузка/Выгрузка), место, дата.

Кнопки **Не принятые**, **В процессе**, **Завершенные** выводят все рейсы с соответствующим статусом в том же формате.

### 3.4 Уведомление водителю о новом рейсе

Отправляется при создании рейса (ручной/1С) и при нажатии «Сохранить рейс» после добавления точек. Текст:
- «У вас новый рейс: \<id\>»
- ТС, прицеп, температура, контакты, номер для регистрации (если заданы)
- Блок «Точки»: тип, место, дата по каждой точке
- Если у водителя уже есть активный рейс — предупреждение и подсказка зайти в «Рейс» позже
- Иначе — инлайн-кнопка «Принять рейс».

### 3.5 Водитель: рейсы

- **Рейс** — вход в меню рейсов.
- **Активный рейс** — если есть рейс в статусе «В процессе»: показ текущей точки (тип, дата, место), кнопки по шагам (Выехал на точку → Зарегистрировался → На ворота → Забрал документы → Выехал с точки). В шапке рейса: номер рейса, ТС, прицеп, температура, контакты, номер для регистрации (если есть). Если все точки выполнены — список точек со статусами и кнопка «Завершить рейс».
- Если активного нет, но есть «новые» рейсы — список новых с инлайн-кнопкой выбора рейса для принятия. В списке по каждому рейсу: id, ТС, прицеп, температура, контакты, номер для регистрации (если есть), количество точек.
- **Принять рейс** (инлайн) — смена статуса рейса на «В процессе», уведомление админам/логистам о смене статуса точки/рейса (в уведомлении — рейс, водитель, ТС, прицеп, температура, контакты, номер для регистрации при наличии).

Уведомления админу/логисту по статусам точек формируются в `services/notification_service.notify_about_status_point`: рейс, водитель, ТС, прицеп, температура, контакты, номер для регистрации, текст статуса точки.

## 4. Зарплата

### 4.1 Администратор / бухгалтер

- **Загрузить расчет за день** — выбор водителя (ФИО) → ввод одной строкой 37 числовых/текстовых параметров расчёта (разделитель — пробел или запятая; десятичные — через точку или запятую). После успешной загрузки — сообщение с деталями расчёта и уведомление водителю.
- **Найти расчет за день** — водитель → дата → вывод расчёта (все ненулевые поля в едином формате отображения).
- **Не подтвержденные расчеты** — список расчётов без подтверждения водителем.
- **Удалить расчет** — водитель → дата → выбор расчёта по ID → подтверждение.
- **Выгрузить расчет за период** — водитель → тип периода (с начала месяца / выбор месяца / произвольный) → при необходимости ввод границ периода → выдача CSV.

**Формат отображения расчёта:** заголовок (ID, дата, рейс при наличии), затем все ненулевые поля (названия полей и значения в едином стиле, включая паллеты, тарифы, адреса, транспорт, прицеп и т.д.), в конце статус (ожидает подтверждения / подтверждено / с комментарием и текст комментария).

### 4.2 Водитель

- **Показать расчет за день** — ввод даты → вывод расчёта за этот день (тот же формат ненулевых полей) и при наличии — инлайн «Подтвердить» / «Комментарий».
- **Не подтвержденные расчеты** — список своих неподтверждённых расчётов с кнопками подтверждения/комментария.
- **Выгрузить расчет за период** — выбор типа периода и границ → выдача CSV.

При подтверждении или отправке комментария бухгалтерам и администраторам уходит уведомление с полным разбором расчёта (все ненулевые поля) через `MessageSalary._format_salary_for_driver` в `notification_service`.

## 5. Заявки на ремонт

### 5.1 Водитель

- **Заявка на ремонт** — номер ТС (форматы как для рейсов) → описание неисправности → подтверждение → создание заявки с ID = MAX(id_ticket)+1, уведомление администраторам и механикам.
- **Мои ремонты** — список своих заявок по статусам.

### 5.2 Администратор / механик

- **Обработать заявку** — выбор заявки из списка (инлайн) → ввод даты, места, комментария → смена статуса, уведомление водителю.
- **Найти заявки ТС** — ввод номера ТС → список заявок по этому ТС.
- Кнопки по статусам: Новые, Ожидают подтверждения, Ожидают ремонта, Выехали на ремонт, Отремонтированы, Завершены — вывод списка заявок в выбранном статусе.

При смене водителем статусов «Выехал на ремонт» и «Завершил ремонт» вызываются `notify_about_departure_for_repair` и `notify_about_completed_repair` — уведомления админам и механикам с полными данными заявки. Подтверждение окончания ремонта — за администратором (инлайн).

**Формат уведомлений по ремонту:** номер заявки, водитель, телефон, ТС, неисправность, дата/место/комментарий ремонта, текущий статус.

---

## 6. База данных: рейсы

Таблица **Route**: id, tg_id, points, status, number_auto, temperature, dispatcher_contacts, **registration_number**, **trailer_number**. Статусы: new, process, success. Для добавления полей «номер для регистрации» и «номер прицепа» выполните миграцию (см. `docs/DB_MIGRATION_ROUTE_EXTRA.md`).

## 7. Где какая информация выводится (сводка)

| Место | Информация о рейсе |
|-------|--------------------|
| Список рейсов (админ/логист) | id, водитель, статус, ТС, прицеп, температура, контакты, номер для регистрации, точки |
| Уведомление водителю о новом рейсе | id, ТС, прицеп, температура, контакты, номер для регистрации, точки |
| Водитель — активный рейс / список новых | id, ТС, прицеп, температура, контакты, номер для регистрации, точки/статусы точек |
| Уведомления админу/логисту по точкам | id рейса, водитель, ТС, прицеп, температура, контакты, номер для регистрации, текст статуса |
| Поиск по водителю / по ТС | тот же формат списка рейсов с фильтром по статусу |
| Ручной ввод рейса | запрос: номер рейса, температура, контакты, номер для регистрации, номер прицепа (все с возможностью пропуска «-») |
| Формат 1С | парсинг номера рейса, номера для регистрации, ФИО водителя, ТС, прицепа, температуры, контактов; при отсутствии — запрос с пропуском |

Логика бота и форматы выводов приведены в актуальном состоянии на момент написания документа.
